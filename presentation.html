Architecture

Coordinator + Specialist split

Ticket arrives │ ▼ [Coordinator] │── get_user_context(email) │ └─ enrich: title, is_csuite, acct status │ │── run_triage(ticket) ←──── [Triage Specialist] │ └─ validation-retry loop (max 3) │ lookup_knowledge │ returns: queue · priority · conf │ get_user_context │ │── _check_escalation(...) ← deterministic │ └─ 6 trigger codes, no LLM │ │── create_ticket(queue, priority, summary) │ │── run_resolver(ticket, triage) ←── [Resolver Specialist] │ └─ auto-resolve if eligible │ lookup_knowledge │ NEVER touches security queue │ resolve_ticket │ └── escalate_to_human / resolve_ticket + structlog JSON for every step

Key rule

Specialists receive only what they need. No implicit context inheritance from the coordinator. Every specialist call is self-contained.

Max 5 tools / specialist

Tool-selection reliability drops past 5. Hard constraint in CLAUDE.md.

All outputs: Pydantic v2

TriageResult, ResolutionResult, CoordinatorOutput — no raw dicts anywhere.

The Brake — Challenge 5

Deterministic escalation rules

Six explicit trigger codes evaluated before any write action. One trigger is sufficient to escalate.

Trigger codeFires whenWhy deterministic, not probabilistic
SECURITY_QUEUEAny ticket routed to securityBlast radius too high — always needs a human
P1_PRIORITYTriage assigns P1Outages move faster than agents can reason
LEGAL_MENTIONGDPR / breach / audit / compliance in bodyLegal exposure cannot be auto-resolved
CSUITE_REQUESTORCEO, CFO, CTO, COO, CISO, VPReputational risk; exec expects human response
LOW_CONFIDENCEClassifier confidence < 0.70Agent uncertainty is itself a signal
RETRY_LIMITRetry count > 2 on validation loopPersistent failures indicate an edge case

"Explicit criteria in place of vague modifiers. 'When the agent isn't sure' is not a rule — it's a wish."

The Tools — Challenge 3

Five tools. Every one teaches its limits.

"""
Tool: resolve_ticket

DOES: Marks an existing ticket resolved,
      stores resolution steps, timestamps closure.

DOES NOT: Perform the actual resolution action.
          Does not notify the requestor.
          Does not close security-queue tickets —
          those must be closed by a human.

Returns structured error on failure:
  {"isError": True,
   "code": "FORBIDDEN",
   "guidance": "Security tickets cannot be
     auto-resolved. Escalate for human review."}
"""

5 tools

lookup_knowledge · get_user_context · create_ticket · resolve_ticket · escalate_to_human

Every failure is structured

{"isError": true, "code": "...", "guidance": "..."}
The agent gets its next move — it never parses a string.

"Does not" is the most important sentence

The LLM uses the docstring to decide when not to call a tool. Vague descriptions produce both over-use and under-use.

The Triage — Challenge 4

Validation-retry loop

for attempt in range(3):
    response = client.messages.create(...)
    try:
        parsed = json.loads(response.content[0].text)
        result = TriageResult(**parsed)   # Pydantic validates
        log.info("triage_success",
                 retry_count=attempt)
        return result
    except (JSONDecodeError, ValidationError) as e:
        # Feed specific error back to Claude
        last_error = {"error": str(e)}
        log.warning("triage_retry", error=str(e))

# After 3 failures: return confidence=0.0
# → triggers LOW_CONFIDENCE escalation

On failure: feed the error back

The specific validation error is added to the message history. Claude sees what it got wrong and self-corrects — not a blind retry.

Max 3 retries

After 3 failures, return confidence: 0.0. This triggers the LOW_CONFIDENCE escalation rule — the agent gracefully degrades instead of guessing.

Every retry is logged

structlog records retry count and error type per ticket. Required for The Scorecard and for post-incident replay.

How We Used Claude Code

What worked · what surprised us

CLAUDE.md first

Writing architecture constraints and tool rules into CLAUDE.md before any code gave every team member consistent scaffolding without a single alignment meeting.

Parallel scaffolding

Claude generated all 5 tool stubs, Pydantic models, and both specialist agents in a single session while BA/PM were writing the Mandate doc. Time to first working pipeline: under 30 minutes.

Adversarial test design

Claude generated prompt-injection and buried-legal-exposure tickets — and caught the C-suite impersonation case (TKT-ADV-005) that the team hadn't thought of.

ADR drafts from CLAUDE.md

Architecture decision records were drafted directly from the conventions in CLAUDE.md. Team reviewed and added the escalation matrix table. Total ADR time: ~15 minutes.

Biggest time save: the validation-retry loop and structured error pattern — 10 minutes to implement correctly once Claude had the full context from CLAUDE.md.

If We Had More Time

Five things, in priority order

1
Wire the PreToolUse hook — block resolve_ticket on security queue at the SDK level. The "hard stop" vs. "slow stop" distinction. Currently only the slow stop (escalation check) is wired.
2
The Loop (Challenge 8) — capture human overrides as labeled examples, feed back into the eval set and as few-shot examples for the triage classifier.
3
Real ServiceNow / AD integration — swap JSON mock data for live CMDB and directory lookups. The tool interfaces are designed for this; it's a config change, not a rewrite.
4
Confidence calibration — the current score is self-reported by the LLM. Post-hoc calibration against eval labels would make the 0.70 threshold meaningful.
5
Expand the eval set — 100+ tickets, even distribution across all six queues and four priorities. Stratified sampling to prevent easy P4s from dominating the accuracy score.
